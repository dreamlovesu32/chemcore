# Chemcore 开发者日志 - 2026-04-26

作者：张家骏

时间范围：2026-04-26 00:00 至 2026-04-26 23:59，Asia/Shanghai

## 总结

今天这轮工作的核心，从键绘制和双键样式，切到了文字编辑器和标签锚点几何本身。前半段的重点是把文字工具从“借浏览器富文本能力”逐步拆成“自有文本模型 + 自有选区 + 自有显示层”的结构；后半段的重点则是把 attached label，尤其是末端双键标签的锚点规则，真正收敛到和渲染几何一致。

这一天里最重要的变化，不是又修掉几个单点偏差，而是把两类长期隐患都往内核里收了一层：

- 文字编辑不再主要依赖 `contenteditable`、DOM 选区和浏览器富文本命令；
- 标签锚点不再靠前台猜测或旧的字号常量，而是尽量对齐内核的 glyph 几何和双键渲染规则。

最终效果上，文字工具现在已经具备更像独立编辑器的形态；而末端双键上的元素标签，也终于回到了用户预期的双线中缝位置，而不是继续停在“看起来差不多但总有点偏”的状态。

## 文字工具内核化

今天一开始做的，不是继续在旧的 `contenteditable` 上补表现问题，而是把编辑器语义逐步从浏览器手里收回来。

这部分工作分了几层。

### 自有文本模型

首先，文字工具的格式操作不再依赖浏览器富文本命令。加粗、斜体、下划线、上标、下标、化学格式、颜色、字号、字体和粘贴，现在都先落到自己的 runs 模型，再由前台重绘显示层。

这意味着：

- 浏览器不再是“真实文档”；
- 真正的文本状态转成了 `sourceRuns + selection`；
- 工具栏命令和提交流程，都开始围绕自有模型工作。

### 自有选区与光标

第二步，是把选区和光标收成显式状态，而不是继续把浏览器选区当唯一真相。

编辑态里现在已经有独立的 `selection` 状态：

- 键盘输入先改 selection/runs；
- 鼠标点选和拖选先更新自己的 offsets；
- 再把结果渲染成蓝色高亮和黑色光标。

换句话说，浏览器原生选区不再驱动编辑器逻辑，而只是逐步退成辅助输入通道。

### 去掉 contenteditable 语义层

后续继续拆到显示层时，编辑器从 `contenteditable` 主导，切成了：

- 可见文本显示层；
- 自绘蓝色选区；
- 自绘黑色光标；
- 隐藏 `textarea` 仅负责承接键盘和输入法。

这里的意义在于，文字编辑的“长什么样”和“怎么改”已经开始脱离 DOM 语义本身。后续要做多端，至少不再被浏览器富文本模型死死绑住。

### 共享 glyph profile

为了减少“编辑时一套、提交后一套”的漂移，今天又把 glyph profile、tracking、上下标缩放和基线偏移继续收成共享数据。

现在前端文字编辑和 Rust 内核的 label/glyph 规则，至少已经开始共用一份 profile 清单，而不是各写一套常量。

这一步虽然还没完全消除所有平台差异，但至少把最容易漂移的那部分参数，先压成了单源。

## 文本编辑器模块拆分

文字工具在功能层面继续补的同时，前台结构也做了一轮较大的拆分。

今天把原本塞在 `viewer/app.js` 里的编辑器逻辑，拆成了几个单独模块：

- `viewer/text_metrics.js`
- `viewer/text_editor_model.js`
- `viewer/text_editor_render.js`
- `viewer/text_editor_controller.js`

拆出去的不只是 helper，而是把下面几类职责分开了：

- 文本长度、字符 advance、glyph profile 等纯度量；
- runs / selection 的模型操作；
- 编辑态显示层渲染；
- 打开会话、输入、导航、指针拖选、格式命令等控制器逻辑。

这样做的直接收益不是“代码更好看”，而是把今天这种持续高频迭代的文字编辑链，先从一个三千多行的单文件里拆出来，降低后续回归成本。

## 化学格式与标签编辑

今天文字工具里和用户交互最相关的几条规则，也继续往正确方向收了。

### 标签默认化学模式

节点/端点标签的输入现在默认走 chemical 语义，目标不是“提交后再重排”，而是编辑态即时上下标。

也就是说，在标签编辑里：

- 元素后的数字会立刻变成下标；
- 电荷和相关数字会立刻变成上标；
- 显示层和提交结果尽量保持同构。

### 已有标签重新编辑

已有标签再次进入编辑时，不再随点击点乱跑，而是优先回到已有标签自己的稳定编辑锚点。  
这件事非常关键，因为它直接关系到“输入时什么样，提交后还是什么样”这条编辑器基本原则。

### 普通文本与标签分轨

今天还继续明确了两条轨道：

- 普通文本框：点击哪里，就从哪里开始；提交后按原位置显示；
- 节点/端点标签：保留吸附、方向判断和 label 规则。

这让普通文本对象和 attached label 不再互相污染逻辑。

## 标签锚点几何修正

今天后半段最关键的一组问题，集中在节点标签，尤其是末端双键标签的锚点上。

### 末端双键锚点规则补齐

前面已经把一部分“末端双键标签应落在两根线中间；一旦有取代基则回主键锚点”的规则补进内核，但今天实际回归后发现，仍然存在两类更隐蔽的问题：

- 某些路径虽然方向对了，但偏移量明显不够；
- 某些单字母标签看起来还没有压在真正中线上。

继续往下查后，问题被分成了两层。

### attached label 自身几何没有严格吃锚点

第一层问题是 attached label 自己的几何还保留着旧的 baseline 近似。  
即使目标锚点是对的，单字母 `O`、`N` 这类标签最终仍可能只“接近锚点”，而不是严格压到锚点上。

今天对此做了修正：

- attached label 先按现有 runs 生成 glyph polygons；
- 再把整组 polygon 和 box 平移到“锚字中心 = 目标锚点”。

这一步把标签自身的字形中心和编辑锚点重新对齐了。

### 双键锚点 offset 和渲染侧不一致

真正更核心的问题在第二层：标签锚点虽然已经开始按末端双键偏移，但所用的双键 offset 公式，和实际渲染双键的公式并不一致。

今天最终定位出的根因是：

- 渲染侧的 side double offset 归一化基准是 `VIEWER_BOND_STROKE = 0.85`；
- 标签锚点侧一开始却在按另一套更大的默认线宽归一化；
- 结果就是标签只偏到了可见双键间距的一小部分，看上去永远“差一点”。

在把这条规则统一之后，斜双键末端的 `O` 标签终于回到了真正的双线中缝，而不是继续停在偏近主键的一侧。

### 方向语义与 begin/end

这轮修正里还顺手收掉了另一个很容易漏掉的问题：  
`Left/Right` 的语义必须跟 bond 自身 `begin -> end` 方向绑定，而不能按“当前端点看向对端”的方向临时解释。否则在 terminal end 端，偏移很容易翻到反侧。

这部分规则也已经纳入内核测试。

## 固定回归图与脚本

为了避免这类几何问题以后再靠肉眼来回确认，今天专门补了一套最小回归脚本。

新增：

- `scripts/label-anchor-regression.mjs`

这套脚本会固定回放三类 attached label 场景，并产出裁剪图：

- 末端 side double：标签应落在两线中间；
- center double：标签应回主键锚点；
- side double 加取代基：标签应回主键锚点。

对应输出在：

- `tmp/label-anchor-regression/terminal-side-double.png`
- `tmp/label-anchor-regression/center-double.png`
- `tmp/label-anchor-regression/branched-double.png`

此外，之前的文字编辑回归脚本也继续保留并反复跑通：

- `scripts/text-editor-regression.mjs`

这让今天这轮修改，不再只是“本地看起来好了”，而是有了固定的脚本化回放基线。

## 风险和下一步

今天虽然把文字工具和标签锚点都收紧了不少，但还有几件事后面值得继续盯：

- 文字编辑现在已经基本摆脱 `contenteditable` 语义层，但输入法、grapheme cluster 和平台级字体差异仍然是后续风险点。
- attached label 的锚点和渲染侧双键 offset 已经重新统一，但后续如果再调 bond 渲染常量，最好继续保证锚点侧和渲染侧共用同一处定义，而不是只保持“数值看起来一样”。
- `viewer/app.js` 虽然已经拆掉一大块，但编辑器控制流仍然是当前前台最复杂的一坨逻辑，后面如果继续长功能，最好坚持模块化推进，而不是再把分支堆回主文件。

今天的结论可以概括成一句话：  
文字编辑器开始像一个真正独立的编辑器了，而 attached label 也终于开始按“内核定义的几何”落位，而不是继续靠浏览器和经验常数凑结果。
