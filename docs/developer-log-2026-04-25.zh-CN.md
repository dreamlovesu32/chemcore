# Chemcore 开发者日志 - 2026-04-25

作者：张家骏

时间范围：2026-04-25 00:47 至 2026-04-25 23:59，Asia/Shanghai

## 总结

今天这轮工作的重点，从键接触几何切到了编辑器交互规则本身。最大的收获不是又多了几个单点修复，而是把 hover 端点改标签、自动标签几何、15° 吸附、双键自动挪边、冻结语义，以及 `Left/Right` 在编辑和渲染之间的对应关系，收成了一套自洽规则。

如果说前一天的核心是“键怎么画”，那今天的核心就是“用户怎么改键、系统怎么自动跟随”。这套规则一旦不统一，就会出现很典型的编辑器错觉：状态里记的是一种双键样式，屏幕上画的是另一侧；或者第一次生成的双键本来应该继续自动调整，却被过早冻结；再或者新标签虽然出现了，但 bond retreat、knockout 和标签锚点马上全部退化。

今天最终把这些问题基本收口到一个稳定状态：hover 改元素/缩写可用，自动标签几何完整，双键样式在未冻结时会按取代基分布自动调整，用户手动接管后又能稳定锁住，而且 `Left/Right` 在状态机和渲染输出里终于重新对应上了。

## Ketcher 键盘映射调研

今天晚些时候又做了一轮和编辑器交互更直接相关的工作，起点不是 Chemcore 自己，而是先去核对了一遍 Ketcher 现有的 hover 端点键盘映射。

这一步的目的很明确：用户要的不是“点击原子后弹出输入框改元素”，而是在画键模式下，鼠标悬停到端点，不点击，直接按键盘把该端点改成元素、缩写或 query 标签。这个交互在 ChemDraw/Ketcher 类产品里是非常高频的，不能靠猜。

调研结论分成两层：

- Ketcher 确实存在 hover 到端点后直接按键替换原子标签的路径；
- 它内部既有一张固定的单键映射表，也有 abbreviation lookup 的动态候选机制。

最终整理出的 Chemcore 侧需求被压缩成一份规则文档：

- `p -> Ph`
- `P -> P`
- `m -> Me`
- `c -> 清回默认碳`

这部分规则单独收在：

- `docs/ketcher-hover-atom-hotkeys.zh-CN.md`

这份文档的价值不在于“抄 Ketcher”，而在于把我们自己的交互口径先钉死。后续如果要继续扩展 `Et`、`Bn`、`Ts` 一类缩写，或者加 query atom 行为，这份文档就是基线。

## Hover 端点直接改元素/缩写

在确认交互口径之后，Chemcore 编辑器新增了“画键模式下 hover 端点直接按键改标签”的路径。

今天做的不是前台 fake overlay，而是直接把这条操作接进 Rust engine：

- viewer 侧在 `bond` 模式拦截 hover 端点键盘输入；
- wasm 暴露新的替换接口；
- engine 直接改当前 hover 节点的标签和原子信息；
- 普通元素、缩写标签和清回碳分别走各自的状态更新逻辑。

最终支持的第一批映射是：

- `p -> Ph`
- `P -> P`
- `m -> Me`
- `c -> C / 清回默认碳`
- 以及一批基础元素键位，例如 `h/n/o/s/f/l/b/i`

其中“清回默认碳”不是只改显示文本，而是把 label 去掉，让节点重新回到默认碳节点的状态。这个细节很重要，因为如果只删文本、不重置原子语义，后续 valence、序列化和样式都会留脏状态。

## 自动生成标签几何

Hover 替换标签接进去以后，前台马上暴露出另一个问题：标签虽然出现了，但 bond retreat、knockout、center double 对标签的避让都退化了。

根因是新生成的标签只有文本内容，没有完整几何。Chemcore 当前的标签避让并不是“看 text 字符串长度差不多估一下”，而是依赖 label 几何字段：

- `label.position`
- `label.box`
- `label.glyphPolygons`

今天把这件事补到了 engine 里：

- 自动生成的标签在创建时就补齐 `position` 和 `box`；
- hover 端点标签被替换后，也会重新刷新节点 label geometry；
- 这样现有的 bond clipping、knockout 和 label retreat 可以立刻复用原有逻辑，不需要另起一套 hover label 专用渲染路径。

这件事的意义在于：标签不是“前台随便写一个文本节点”，而是文档模型的一部分。只要文档里存在 label，它就必须有足够的几何信息，让后端渲染规则能继续工作。

## 标签字号调整

新生成的缩写标签在几何补齐后，视觉上又暴露出第二个问题：默认字号偏小，和已有结构标签不协调。

今天针对自动生成标签的默认字号做了两次上调，最终把默认值拉到 `15`。这部分改动虽然看上去只是 UI 微调，但其实和几何关系很紧：

- 字号变了，label box 跟着变；
- label box 变了，键退让和 knockout 也跟着变；
- 所以这不是一个“纯前台 CSS”问题，而是编辑器文档默认值和渲染几何的同步问题。

## 角度吸附改成 15°

今天还顺手把画键的角度吸附从原来比较稀疏的一组角，改成了完整 `15°` 步进。

这里改的不是某个前台拖拽 helper，而是编辑器吸附口径本身：

- 全局吸附角集改成 `0°..345°` 每 `15°` 一档；
- 相对已有键的候选角集也改成 `15°..180°` 每 `15°` 一档。

最终效果是：

- 水平、垂直、`30°/45°/60°/75°` 等常见化学绘图角度都能自然吸附；
- 端点点击延伸和拖拽延伸走的是同一套角度集合；
- 后续如果要继续对齐 ChemDraw 风格模板角度，这套基础离散角已经够用了。

## 双键与标签：长度、间距和锚点

今天晚上的另一块重点，是把“双键与标签”的规则真正收敛到用户能接受的口径。

一开始暴露的问题有两个：

- 标签出现后，双键副线间距会变窄；
- 某些等长双键在带标签端点时，两根线看上去不再平行。

这两个问题的根因并不一样。

### 标签不应改变真实 bond 长度

用户明确指出：标签 retreat 不是 bond 自身“缩短了”，只是视觉上在端点附近让开文字而已。除了 wedge 家族以外，键的实际长度和多重键间距计算，都应该基于真实节点中心，不应该基于剪裁后的可见线段。

今天据此修正了：

- side double / triple 的 offset 计算，改回基于实际节点坐标；
- side inset 的长度尺度也改回基于真实 bond length；
- label clipping 只负责视觉退让，不再影响多重键间距缩放。

这个口径非常关键。否则一旦标签较宽、端点剪裁较多，副线间距就会被错误压扁，看起来像 bond style 自己变了。

### 等长双键不平行的根因

另一处 bug 更隐蔽：中心双键本身已经按中心线对标签做过一次退让，但两条子线在后续 profile / boundary line 计算中，仍然有部分地方继续按另一套长度基准单独求 offset，导致两条线的端帽和 join profile 挂到了不完全一致的平行线上。

这会表现成：

- 线长可能看着差不多；
- 但两条线的方向微妙偏开；
- 最终视觉上像是“不平行”。

这块今天做了两轮修正：

- 第一轮先统一中心双键绘制和 endpoint profile 的 offset 基准；
- 第二轮进一步把 label 参与后的局部二次 clipping 关掉，避免中心双键子线在标签端各自再吃一遍不同的 retreat。

修正后的规则是：

- 中心双键的两条线共享同一中心线退让结果；
- 不再让子线在 label 端各自跑出不同几何；
- 所以视觉上重新保持真正平行。

### 末端双键与标签锚点

用户还明确补充了一个以前没收清楚的规则：

- 末端 side double / center double 在没有继续取代时，标签应该锚在两根线的中间；
- 一旦该端再接出取代基，就回到普通主键锚点逻辑。

这件事也接到了自动生成标签的 geometry 刷新路径里。换句话说，标签锚点不再是“写死跟着节点中心”，而是会看当前节点上的双键 placement 和连接数。

## 双键自动样式与冻结语义

今晚最反复的一块是双键在编辑阶段的自动样式规则。这里不是单纯的渲染问题，而是“什么时候自动挪边，什么时候不该再动”的编辑器状态机问题。

### 初始默认样式

用户要求的起始口径是：

- 当一根键被首次改成双键，而两端已有较多连接，或者本来就在多连接场景里生成双键，默认先给 `center double`；
- 这是因为在歧义场景下，center double 比任意单边双键更中性，也更容易再根据后续取代自动分流。

这条规则今天已经接到：

- 点击现有 bond 中心、用双键工具首次改成双键；
- 用双键工具直接新画双键；
- 用虚实双键工具新画或首次生成 dashed double。

### 第三根键、第四根键与自动挪边

自动挪边的规则后来被用户重新定义得更清楚了：

- 只要双键还没有冻结，任何后续加取代基都允许自动调整 placement；
- 二取代双键再画第三根键时，双键挪到取代更多的一侧；
- 如果再画第四根键，出现两侧打平，则双键挪到最后一根新键落笔的那一侧；
- 单取代双键在另一端继续画，如果形成顺式关系，则双键自动落到“内侧”。

这套规则最终没有拆成很多互相覆盖的小特例，而是尽量收成一套“按 bond 轴线统计相邻取代落在左右哪一侧”的逻辑。这样第三根键、第四根键和单取代顺式补键这三类 case 才能共享同一个判定框架。

### 冻结语义

真正棘手的不是 placement 算法，而是“何时冻结”。

用户的最终定义是：

- 只要这根双键是第一次诞生出来的，不管是由单键改来、由别的样式改来，还是直接新画出来，都不冻结；
- 只有当用户再次单击一个已经存在的双键，手动切换过它的样式，才视为用户显式接管，此后冻结；
- 冻结后，后续再画取代基，不再自动改样式；只能用户自己再点双键调整。

今天中间出过一次偏差：一开始把“任何点击后形成的双键”都标记成了 `frozen`，结果导致“首次改成双键”也被错误锁死。后来又进一步收窄成：

- 只有这根 bond 原本就已经是双键时，再点才冻结；
- 所有首次生成双键的路径都保持 `frozen = false`。

这部分最后在 `DoubleBond` 上新增了显式 `frozen` 字段，并给所有创建/循环样式路径都补了初始化和测试。

## Left / Right 侧别最终校正

今天最容易让人误判的一处问题，是“内侧到底是不是搞反了”。

一开始在水平 case 上，自动挪边逻辑看起来像是对的；但一旦换成斜双键，用户马上发现所谓“内侧”落到了视觉外侧。最后定位出来，问题并不在自动选边算法本身，而在于：

- 编辑逻辑里 `DoubleBondPlacement::Left/Right` 的含义是一套；
- 渲染层把 `Left/Right` 画到了反侧；
- 自动生成标签的双键中线锚点，也沿用了这套反向映射。

于是就出现了这种错觉：

- 状态机里记录的是 `Left`；
- 测试里读出来也是 `Left`；
- 但屏幕上实际画出来在另一边。

最终修正不是继续改编辑算法，而是把渲染映射和标签锚点映射统一回编辑侧的语义：

- side double 渲染时 `Left/Right` 对应的 offset side 全部翻正；
- outer bond candidate side 和 fragment offset 选择一起改；
- 自动生成标签在 side double 上的偏移方向也跟着统一。

这一步之后，“placement 是什么”和“屏幕上画在哪一侧”终于重新一一对应。

## 今日最终结果

到今天结束时，这一轮工作的结果可以概括成几条非常具体的编辑器行为：

- 画键模式下 hover 端点可以直接用键盘改成元素或缩写；
- 自动生成标签有完整几何，不会再把现有 retreat / knockout 逻辑打穿；
- 新标签的字号和现有化学标签更接近；
- 画键角度吸附已经收成完整 `15°` 步进；
- 标签 retreat 不再错误影响双键/三键的真实长度与副线间距；
- 末端带标签的中心双键重新保持平行；
- 双键首次诞生时不冻结，用户再次手动改单击样式后才冻结；
- 未冻结双键会在后续加取代基时按取代分布自动挪边；
- `Left/Right` 在编辑、渲染、标签锚点三层已经重新统一。

这几件事放在一起，解决的其实不是若干独立小 bug，而是把“用户对双键的直觉”和“文档模型里记录的双键状态”重新拉回到一致的系统里。
